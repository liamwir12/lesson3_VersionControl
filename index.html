Version Control

Version control is a logging system that archives any alterations to a file or set of files over a period of time so that you can access them later. These systems allow you to revert files back to a previous state or version, compare changes over a period of time, return the project back to a previous state, see what was last modified and who completed that modification (also revealing something that might be causing a problem), and much more. Utilizing a Versioning Control System (VCS) also helps the user if they mess anything up or lose any of the files. Files can easily be recovered and all versions accessible.

History of Git

Inspired by the Linux kernel, an open source software project of a fairly large scope, git is a free open source versioning control system that stores information. After BitKeeper, a patented distributed version control system (DVCS) broke down, and the tool’s free-of-charge status was revoked the Linux development community developed their own tool basing their design on some of the principles they realized through BitKeeper, such as speed, simple design, fully distributed, and able to handle projects with large data size, among other things. Git has transformed and adapted to be easily accessible, while also following their initial principles that the system was built on.

How Does Git Work

While most VCS store information as a list of file-based changes, and think of the information they store as a set and the changes made to each file over time, Git differs in the sense that when you commit, the file’s state is recorded similarly to taking a screenshot on your phone. In the name of efficiency, Git only stores the file if it has changed. If the file remains the same, a link to the previous identical file. Rather than working like a normal VCS, Git operates more like a filing system with additional tools to help it, making it more powerful and effective in many cases.

Getting a Git Repository

Typically, you can gain a Git repository by taking a local directory that is currently not under version control, and turn it into a Git repository, or duplicate an existing Git repository from somewhere else. Project directories that are not presently under version control that you want to start controlling with Git, need the user to go to that project’s directory and type “$ git init”. By completing this step, the user creates a new subdirectory named “.git” that contains all of their necessary repository files. If the user needs to start version-controlling already existing files they should begin tracking those files by doing an initial commit. By typing “$ git commit”.

Cloning an Existing Repository

If the user needs a copy of an existing Git repository the command they would need to type is “git clone”. Many VCS systems use the command "clone" and not "checkout", which is important to note because instead of getting just a working copy, Git receives a full copy of nearly all the data that the server has. When you run the “git clone” command, every version of every file for the history of the project is pulled from the repository and is often taken advantage of when a user’s service disk is corrupted. The user should be able to access nearly any of the clones on any client to set the server back to the state it was in when it was duplicated.
